# MoaIMF v0.1 설계 명세

- 작성일: 2026-06-21
- 상태: v0.1 구현 완료·릴리스 검증 대기
- 라이선스: MIT
- 제품 문구: **Initial. Medial. Final. Composed.**

## 1. 목적

MoaIMF는 사용자가 지정한 macOS 폴더를 감시하여 파일명과 폴더명을 Unicode NFC로 정규화하는 메뉴 막대 앱이다.

해결하려는 핵심 문제는 macOS 및 외부 저장장치·NAS·압축 해제·다운로드 도구를 거치며 생성된 NFD 한글 파일명이 NFC 검색어와 일치하지 않는 현상이다.
이 차이로 Alfred, Spotlight, 터미널 명령 및 경로 기반 자동화에서 파일이 누락될 수 있다.

MoaIMF는 파일 내용을 변경하지 않는다. 이름 변경이 안전하다고 판단된 항목의 마지막 경로 구성요소만 NFC로 바꾼다.

## 2. v0.1 범위

### 포함

- macOS 13 이상을 지원하는 네이티브 SwiftUI 메뉴 막대 앱
- 첫 실행 시 `~/Downloads`를 기본 감시 후보로 제시
- 설정 창의 `+`와 `-`를 통한 감시 폴더 추가·제거
- 등록 폴더의 파일과 하위 폴더 재귀 검사
- FSEvents 기반 변경 감지
- 앱 시작·스트림 복구·저빈도 정기 조정 검사
- 다운로드·파일 쓰기 안정화 판정
- 기본·사용자 정의 안정화 예외 규칙 관리
- NFC 후보와 충돌의 읽기 전용 사전 검사
- 충돌 없는 파일명과 폴더명의 NFC 변경
- 외장 볼륨 분리·재연결과 접근 권한 상태 표시
- 감시 일시정지·재개, 즉시 검사, 앱 종료
- 로그인 시 자동 실행 설정
- 최근 변경·충돌·오류 기록
- 로컬 전용 처리

### 제외

- 파일 내용의 문자 인코딩 변경
- NFC 이름을 NFD로 되돌리는 기능
- 충돌 항목의 자동 이름 생성 또는 덮어쓰기
- `.app`, `.bundle`, `.framework`, `.photoslibrary` 등 패키지 내부 변경
- 심볼릭 링크 대상 경로 순회
- Spotlight 전체 재색인 자동 실행
- 클라우드 동기화, 사용자 계정, 텔레메트리
- App Store 배포와 자동 업데이트
- macOS 12 이하 지원

## 3. 제품 동작

### 3.1 첫 실행

1. 앱은 `~/Downloads`를 첫 감시 후보로 표시한다.
2. 사용자가 폴더 접근을 승인하면 읽기 전용 전체 검사를 실행한다.
3. 기존 Non-NFC 항목 수와 충돌 수를 보여준다.
4. 사용자가 적용을 승인한 후에만 기존 항목을 변경한다.
5. 이후 생성되거나 변경되는 안정화된 항목은 자동 처리한다.

접근 승인을 거부하면 앱은 실행 상태를 유지하되 `권한 필요`를 표시한다. 승인 전에는 이름을 변경하지 않는다.

### 3.2 감시 폴더 추가와 제거

- `+`는 `NSOpenPanel`을 열어 폴더를 선택한다.
- 추가 직후 기존 항목을 읽기 전용으로 검사하고 결과를 보여준다.
- 기존 항목 변경은 별도 확인을 받은 뒤 실행한다.
- 새로 생성되는 항목의 자동 처리는 폴더 등록과 함께 활성화한다.
- `-`는 감시와 향후 처리를 중단할 뿐, 이미 변경한 이름을 되돌리지 않는다.
- 폴더별 활성화 스위치와 `지금 검사` 동작을 제공한다.
- 같은 폴더의 중복 등록을 차단한다.
- 등록된 상위 폴더에 포함되는 하위 폴더와 그 반대 경우를 차단하고 이유를 표시한다.
- 감시 폴더가 없으면 오류가 아닌 `감시할 폴더 없음` 상태가 된다.

### 3.3 사용자 정의 안정화 예외 규칙

- 앱은 `.crdownload`, `.download`, `.part`, `.partial`, `.tmp`를 기본 규칙으로 제공한다.
- 기본 규칙은 설정 화면에 잠금 상태로 표시하며 삭제하거나 비활성화할 수 없다.
- 사용자는 설정 화면의 `+`와 `-`로 사용자 규칙을 추가·삭제할 수 있다.
- 사용자 규칙 종류는 `정확한 파일명`, `접미사/확장자`, `글로브` 세 가지다.
- 글로브는 마지막 경로 구성요소에 대한 `*`와 `?`만 지원하며 정규표현식과 경로 구분자는 허용하지 않는다.
- 모든 패턴은 1~255자이며 NUL과 `/`를 허용하지 않는다. 접미사/확장자 규칙은 `.`으로 시작해야 한다.
- 정확한 파일명 규칙에서는 `*`와 `?`를 문자로 사용할 수 없고, 이를 포함하려면 글로브 규칙을 선택해야 한다.
- 규칙은 모든 감시 폴더에 공통으로 적용하고 재실행 후에도 유지한다.
- 같은 종류와 패턴의 사용자 규칙은 중복 등록하지 않는다.
- 비교 전에 규칙과 실제 이름을 NFC로 정규화하고, 대소문자 비교는 대상 볼륨의 특성을 따른다.
- 일치한 파일이나 폴더는 자동 rename 대상에서 제외한다.
- 수동 미리보기와 기존 항목 적용도 같은 규칙을 따르며 v0.1에서는 일회성 우회 기능을 제공하지 않는다.
- 일치 항목은 다운로드가 진행 중일 가능성이 있는 것으로 취급하므로 해당 항목이 사라지거나 규칙이 제거될 때까지 상위 폴더 rename도 보류한다.
- 일치한 폴더는 내부를 순회하지 않는다.
- 규칙을 추가하거나 삭제하면 활성 감시 폴더의 조정 검사를 즉시 예약한다.
- 미리보기와 감시 상태에는 어떤 규칙 때문에 항목이 보류됐는지 표시한다.

### 3.4 메뉴 막대

메뉴 막대는 다음 상태를 구분한다.

- 감시 중
- 검사 중
- 일시정지
- 주의 필요

메뉴의 기본 동작은 다음과 같다.

- 감시 일시정지 또는 재개
- 지금 모두 검사
- 감시 폴더 설정 열기
- 최근 기록 열기
- 로그인 시 자동 실행 전환
- MoaIMF 종료

`일시정지`는 앱과 메뉴 막대 아이콘을 유지하지만 이벤트 처리와 정기 검사를 중단한다. 일시정지 상태는 앱 재실행 후에도 유지한다.

`종료`는 진행 중인 검사를 협조적으로 취소하고 앱 프로세스를 끝낸다.
별도 daemon, LaunchAgent 또는 helper는 남기지 않는다.
로그인 시 자동 실행이 켜져 있다면 메뉴에 `다음 로그인 시 다시 실행됨`을 명시한다.

### 3.5 알림

정상적인 이름 변경에는 시스템 알림을 보내지 않는다. 다음 상황에서만 알림을 보낸다.

- 충돌로 변경하지 못한 항목 발생
- 감시 폴더 접근 권한 상실
- 감시 볼륨 연결 해제 또는 재연결 실패
- 반복되는 파일시스템 오류

## 4. 안전 정책

### 4.1 처리 대상

- 일반 파일과 폴더의 이름을 처리한다.
- 숨김 파일과 숨김 폴더도 처리한다.
- 패키지는 하나의 항목으로 취급하며 내부는 순회하지 않는다.
- 심볼릭 링크는 링크 이름만 처리하며 대상을 따라가지 않는다.
- 앱이 관리하는 상태·로그 경로는 감시 대상에서 제외한다.

### 4.2 다운로드 안정화

다음 조건을 모두 만족해야 자동 변경 후보가 된다.

- 기본 또는 사용자 정의 안정화 예외 규칙과 일치하지 않는다.
- 파일 식별자, 크기, 수정 시각이 최소 30초 간격의 두 검사에서 동일하다.
- 처리 직전 같은 상태를 다시 확인한다.
- 폴더는 하위 트리에 불안정한 항목이 없을 때만 이름을 변경한다.

상태 추적 키는 문자열 경로보다 `fileResourceIdentifier`를 우선한다. 파일시스템이 안정적인 식별자를 제공하지 않으면 정규화된 볼륨 식별자와 상대 경로를 보조 키로 사용한다.

### 4.3 충돌 검사

대상 폴더의 실제 형제 이름을 수집한 뒤 NFC 결과별로 그룹화한다. 대상 볼륨이 대소문자를 구분하지 않으면 비교 키에 대소문자 접기를 적용한다.

둘 이상의 실제 항목이 같은 목표 이름으로 모이면 해당 항목을 변경하지 않는다. 어떤 경우에도 기존 항목을 덮어쓰거나 자동으로 접미사를 붙이지 않는다.

APFS에서 `exists()`는 현재 NFD 항목을 NFC 목표 경로에서도 존재하는 것처럼 반환할 수 있으므로 충돌 판정 근거로 사용하지 않는다.

### 4.4 이름 변경

1. 전체 변경 계획과 충돌을 먼저 계산한다.
2. 경로 깊이가 깊은 항목부터 처리한다.
3. 원본 항목의 식별자와 상태를 다시 확인한다.
4. NFD 이름을 NFC 이름으로 직접 변경한다.
5. 디렉터리 항목을 다시 읽어 실제 저장 이름이 NFC인지 확인한다.
6. 직접 변경이 실패하거나 실제 이름이 바뀌지 않은 파일시스템에서만 고유 임시 이름을 이용한 2단계 변경을 시도한다.
7. 2단계 변경 전 복구 저널을 기록하고, 중간 실패 시 원래 이름으로 복구한다.

Foundation의 file URL과 `FileManager`는 한글 경로를 파일시스템 표현으로 변환하면서 NFC 마지막 구성요소를 NFD로 바꿀 수 있다. 따라서 저장된 raw 이름 검증은 POSIX `readdir`의 `d_name` 바이트로 수행하고, 2단계 변경의 최종 목적지는 NFC `targetName.utf8`을 `renameatx_np`에 직접 전달한다. URL은 권한·부모 경로·표시용으로 사용하되 raw 이름의 진실 공급원으로 사용하지 않는다.

모든 변경은 사전 충돌 검사와 덮어쓰기 금지 조건을 만족해야 한다.
사후 검증에서 NFC가 보존되지 않는 볼륨은 `지원되지 않는 정규화 동작` 상태로 표시하고 자동 처리를 중단한다.

### 4.5 기록과 개인정보

- 처리 기록은 사용자 계정의 Application Support 아래에 저장한다.
- 기본 보존 기간은 30일이다.
- 기록에는 시각, 결과, 이전·새 경로, 오류 사유를 포함한다.
- 네트워크로 데이터를 전송하지 않는다.
- 텔레메트리와 분석 SDK를 포함하지 않는다.

## 5. 감시와 조정 검사

FSEvents는 완전한 작업 목록이 아니라 재검사 신호로 사용한다.

`새 항목만 감시`를 선택한 루트는 등록 시점에 존재하는 항목의 안정적인 파일 식별자를
기준선으로 저장한다. 시작·재개·정기 조정 검사와 `MustScanSubDirs` 전체 검사는 이 기준선을
자동 변경 대상에서 제외한다. 이후 생성되거나 이동된 항목은 이벤트 경로 검사와 조정 검사에서
처리하며, 사용자가 명시적으로 실행한 미리보기와 적용은 기준선을 우회한다. 폴더를 다시 선택하면
새 위치의 기존 항목으로 기준선을 갱신한다.

- 활성 감시 루트를 하나의 이벤트 모니터가 관리한다.
- 이벤트는 짧은 시간 동안 합쳐 중복 검사를 줄인다.
- 이벤트가 누락되거나 `MustScanSubDirs`가 발생하면 해당 루트를 전체 검사한다.
- 앱 시작 시 모든 활성 루트를 조정 검사한다.
- 감시가 활성화된 동안 저빈도 정기 조정 검사를 실행한다.
- 외장 볼륨 연결 상태 또는 bookmark가 바뀌면 이벤트 스트림을 재구성한다.
- 앱 자체 rename이 다시 이벤트를 발생시켜도 NFC 상태 검사는 멱등적으로 종료한다.

정기 조정 검사의 기본 간격은 6시간이다. 안정화 대기 후보가 있으면 해당 후보만 30초 후 다시 검사한다.

## 6. 권한과 수명주기

### 6.1 폴더 권한

- App Sandbox를 활성화한다.
- `com.apple.security.files.user-selected.read-write`와 앱 범위 security-scoped bookmark 권한만 요청한다.
- 네트워크 client·server entitlement는 요청하지 않는다.
- 사용자가 선택한 폴더는 security-scoped bookmark로 저장한다.
- 접근 시작과 종료 호출은 균형을 맞춘다.
- stale bookmark는 자동 갱신을 시도한다.
- 권한 취소, 폴더 이동 또는 볼륨 재포맷으로 복원이 불가능하면 `다시 선택 필요`를 표시한다.
- 문자열 경로뿐 아니라 canonical URL과 파일 식별자로 중복을 판정한다.

### 6.2 로그인 실행

- `SMAppService.mainApp`을 사용한다.
- 앱 내부 상태와 시스템 설정의 실제 승인 상태를 함께 표시한다.
- 상태는 `활성`, `승인 필요`, `비활성`, `찾을 수 없음`으로 구분한다.
- 승인이 필요하면 시스템 설정의 로그인 항목 화면을 여는 동작을 제공한다.
- 앱 종료와 로그인 실행 설정은 서로 독립적이다.

## 7. 아키텍처

MoaIMF는 하나의 앱 프로세스로 구성한다.

- `AppUI`: `MenuBarExtra`, 설정, 기록과 권한 복구 화면
- `WatchedFolderStore`: bookmark, 폴더 상태, 정규화 모드와 기존 항목 기준선, 중복·중첩 검사
- `EventMonitor`: FSEvents 등록과 스트림 재구성
- `ScanCoordinator`: actor 기반 직렬 실행, debounce, 취소, 조정 검사
- `StabilityTracker`: 항목 식별자와 안정화 스냅샷
- `StabilityRuleStore`: 기본·사용자 정의 예외 규칙 검증과 영속화
- `NormalizationEngine`: 재귀 스캔, 충돌 분석, bottom-up rename, 사후 검증
- `HistoryStore`: 최근 변경, 충돌, 오류와 복구 저널
- `LoginItemManager`: `SMAppService` 상태와 설정

외부 패키지 없이 Apple 플랫폼 프레임워크만 사용한다.

## 8. 배포

v0.1은 GitHub 공개 저장소에서 MIT 라이선스로 개발한다.

초기 배포 단계는 다음과 같다.

1. 소스에서 직접 빌드할 수 있는 상태를 제공한다.
2. 자동화된 테스트와 재현 가능한 Release 빌드를 마련한다.
3. 서명·공증 환경이 준비되면 GitHub Releases에 앱 번들을 제공한다.

App Store 배포, Sparkle 기반 자동 업데이트와 별도 패키지 관리자는 v0.1 범위에서 제외한다.

## 10. 완료 기준

v0.1은 다음 조건을 모두 만족할 때 완료로 본다.

- 충돌 없는 안정화된 Non-NFC 이름을 NFC로 변경한다.
- 파일 내용과 확장 속성을 변경하지 않는다.
- 충돌이 있을 때 기존 항목을 덮어쓰지 않는다.
- 사용자가 승인하지 않은 폴더를 순회하지 않는다.
- 다운로드와 압축 해제 중인 항목을 조기에 변경하지 않는다.
- 기본 규칙과 사용자 정의 안정화 예외 규칙을 일관되게 적용하고 사용자 규칙을 재실행 후 복원한다.
- 감시 폴더 추가·제거와 권한 복구가 재실행 후에도 유지된다.
- 일시정지와 종료 동작이 UI 설명과 일치한다.
- 종료 후 숨은 helper 또는 감시 프로세스가 남지 않는다.
- 로컬 테스트, 정적 검사, 빌드와 실제 메뉴 막대 수동 QA를 통과한다.
- README와 릴리스 문서가 실제 제공 기능과 일치한다.
